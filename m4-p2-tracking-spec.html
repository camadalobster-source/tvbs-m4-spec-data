架構 — 為什麼 ID 是一切的前提

🎯
目的：為了讓所有追蹤工具共用同一張身分證 (mpid)，使 Web 和 App 的使用者數據能跨平台歸戶比對。如果第一筆事件在 mpid 取得前就發出，該筆數據就會「斷頭」——無法歸戶。因此 ID Sync SDK 必須在所有追蹤標籤之前完成載入。

🌉 什麼是 Bridge？為什麼 GA4 和 mParticle 都要做？

Bridge 是 App 開發者預先在 WebView 裡注入的「通訊通道」。Web 前端透過 JavaScript 偵測通道是否存在——如果存在代表自己在 App 內，就把事件轉交給 App 統一發送。GA4 和 mParticle 各自需要一套獨立的 Bridge，兩者互不影響。

Web 前端：config 加一行 bridge name · App 端：初始化時綁定同名 Bridge · SDK 自動處理：偵測環境 + 切換中介模式

❓ 常見問題 Q&A

▶mParticlemParticle WebView Bridge 實際上要做什麼？有官方文件嗎？

白話說明：mParticle 的 Bridge 不是完全自動的，工程師有兩件事要做。但比起 GA4 的 Bridge（完全自建），mParticle 的機制簡單很多——設定完 bridge name 後，SDK 會自動偵測環境和切換模式。

Web 前端要做：

在 mParticle SDK config 中加入一行設定：
window.mParticle.config.requiredWebviewBridgeName = 'tvbsNewsbridge';

App 端要做（iOS / Android 各自）：

在 WebView 初始化時，照官方文件綁定同名的 Bridge。App 端需要呼叫 mParticle 的 WebView Bridge API，將 bridge name 設為 tvbsNewsbridge。

設定完成後 SDK 自動處理的事：

Web SDK 會自動偵測是否在 WebView 內 → 如果是就自動切換為「中介模式」→ 事件轉交 Native SDK 統一發送 → 不會重複。工程師不需要自己寫 if (window.xxx) 判斷。

⚠ 關鍵：Web 與 App 的 bridge name 必須完全一致（tvbsNewsbridge），大小寫也要相同，否則 SDK 偵測不到。

📎 官方文件：

• mParticle Web SDK — WebView Bridge 文件
• mParticle iOS SDK — WebView Bridge 設定
• mParticle Android SDK — WebView Bridge 設定

▶ID Syncmpid 沒拿到（5 秒逾時）怎麼辦？各工具會怎樣？

結論：mpid 沒拿到 = 該筆事件失去「跨工具歸戶」能力，但每個工具各自的基礎量測和網站功能完全不受影響。

工具mpid 未取得時的行為影響程度
GA4照送事件。mpid User Property 帶空字串 ""。事件進報表，流量數字不少。該筆無法跨工具歸戶
mParticle照送事件。SDK 用內部匿名 ID 追蹤。該筆無法跟 GA4 對照
Comscore完全不受影響。不依賴 mpid。無
Clarity完全不受影響。不依賴 mpid。無

關於 5 秒等待時間：

• 5 秒是業界常見的平衡點，不是官方硬性規定
• 等待的是 Promise（非同步），不會阻塞頁面渲染，使用者感受零影響
• 正常網路環境下 SDK 約 1-2 秒就緒，99% 不會真的等到 5 秒
• 太短（1-2 秒）→ 慢速網路下太多 undefined；太長（10+ 秒）→ 使用者可能已換頁
• 不影響 Core Web Vitals（非同步 Promise 不構成 Long Task）

▶GA4ID Sync 為什麼不能直接把 mpid 送給 GA4？為什麼中間還需要前端處理？

⚠ 以下為 Claude 的分析建議，實際架構請由 Ted 和 Ethan 討論決定

背景：GA4 是 Google 的產品，它透過 GTM DataLayer 或 Firebase SDK 接收資料。ID Sync SDK 是 mParticle 體系，兩者之間需要一個橋接步驟。

比喻：ID Sync 像是「發卡機」發出悠遊卡（mpid），但要搭高鐵（GA4）還是得到高鐵的閘門（GTM）刷一次。

目前設計的流程：

步驟負責方做什麼
① 產出 mpidID Sync SDK（Ted 團隊維護）初始化 mParticle → 取得 mpid → 存在 Promise 裡
② 橋接推送前端 RDawait Promise 拿到 mpid + 讀取 tvbs_uid → dataLayer.push 推給 GTM
③ 設定歸戶GTM / GA4（Ted 團隊設定）收到 mpid_ready → Config Tag 啟動 → 把 mpid + user_id 寫入 User Properties

追問：dataLayer 是全域的，為什麼 SDK 不直接推？

技術上可行——dataLayer 就是一個全域 JavaScript 陣列，任何程式碼都能往裡面塞。Claude 認為分開處理可能有以下好處，但最終由 Ted 和 Ethan 判斷：

① 耦合考量：如果 SDK 內部直接推 dataLayer，SDK 就必須知道 GTM 的事件名稱和變數格式。兩邊改動時可能互相影響。

② 控制權考量：前端除了推 mpid，還要同時推 user_id（tvbs_uid Cookie）。Cookie 的讀取邏輯是否應該放在 SDK 內，需要討論。

③ 可測試性考量：前端 await Promise 再推 dataLayer，這個流程可以在前端斷點除錯。

💬 待討論：第 ② 步由前端橋接是否為最佳做法，或是否有更好的整合方式，請 Ted 和 Ethan 在 Refinement 中確認。
