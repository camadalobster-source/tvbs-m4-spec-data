架構 — 為什麼 ID 是一切的前提

核心問題：TVBS M4 同時使用五個追蹤工具，其中三個（mParticle、GA4、Comscore）需要共用同一張「身分證」(mpid)。如果使用者的第一筆瀏覽事件在 mpid 尚未取得前就發出，該筆數據就會「斷頭」——無法歸戶、無法跨平台比對。ID Sync SDK 的工作就是在所有追蹤標籤之前，先把身分證準備好。

架構 — Web / App / WebView 環境對照

同一套追蹤邏輯，在三種環境下的實作方式不同。WebView 是最複雜的情境——App 內開啟網頁時，Web 端的 SDK 和 App 端的 SDK 會同時存在，若不加以處理，事件會重複發送。因此需要 Bridge（橋樑）機制：讓 Web 端的 SDK 偵測到自己在 App 內，就不自己送事件，改為把資料「轉交」給 App 端的 Native SDK 統一發送。

🌉 什麼是 Bridge？為什麼需要兩套？

Bridge 是 App 開發者預先在 WebView 裡注入的「通訊通道」。Web 前端透過 JavaScript 偵測這個通道是否存在，如果存在就代表自己在 App 內，就把事件資料透過通道傳給 Native SDK，由 Native 統一發送。這樣就不會 Web 送一次、App 又送一次。
GA4 Bridge（自建）

Android：AnalyticsWebInterface（addJavascriptInterface）
iOS：firebase 通道（WKScriptMessageHandler）
偵測到 Bridge → page_view 改名 screen_view + display_mode=app_webview

mParticle Bridge（官方內建）

設定：requiredWebviewBridgeName: 'tvbsNewsbridge'
注意：Web 與 Native 的 bridge name 必須完全一致
兩套 Bridge 獨立運作、互不影響

ID Sync SDK — 全站身分識別核心

ID Sync SDK 的唯一目的：收攏 Web 端的 Identity，確保所有追蹤事件都帶著正確的身分證 (mpid)。它內部包裹了 mParticle SDK，負責初始化並產生 mpid。一旦 mpid 就緒，GA4、mParticle Screen View、Comscore 等工具才能正確歸戶。
為什麼必須 Hard-code？
1. 防禦無痕模式
Ad-blocker 可能攔截 GTM，一旦 GTM 被擋，身分識別機制就整個停擺
2. 時序保證
確保在任何 GA4 或行銷標籤觸發前，mpid 已就緒，不會出現「數據斷頭」
3. 基礎設施等級
定位等同網站核心 CSS / JS，不可選擇性載入或延遲載入

🔗 Promise 橋樑機制（推薦實作方式）

核心概念：建立全域 window.mParticleMpIdReadyPromise，所有下游模組（GA4、mParticle Screen View）await 同一個 Promise 即可取得 mpid
onReady callback：SDK 完成後觸發，但不帶參數——必須在 callback 內主動呼叫 getMPID() 取值再 resolve
逾時防呆：5 秒未就緒（如被 Ad-blocker 攔截）→ resolve(undefined)，避免整站追蹤卡死
驗收標準 (DoD)
  • 1Console 執行 await window.mParticleMpIdReadyPromise → 回傳 mpid 字串（如 "1234567890123"）
  • 2開啟 Ad-blocker 重整 → 5 秒後 resolve undefined，頁面功能與排版不受影響
  • 3網頁原始碼中，ID Sync 的程式碼必須排在 GTM 的程式碼前面（瀏覽器從上往下執行，誰在前誰先跑）

🔀 App 原生 vs WebView — GA4 分流重點

App 原生畫面（首頁、分類、會員等）：由 Firebase SDK 直接發送 screen_view，display_mode = app_native
App 內 WebView（文章頁、搜尋頁等）：Web 前端偵測 JS Bridge 存在 → 將 page_view 改名 screen_view，display_mode 覆寫為 app_webview → 呼叫 Native Bridge 發射給 Firebase
Bridge 偵測：Android 檢查 window.AnalyticsWebInterface · iOS 檢查 window.webkit.messageHandlers.firebase
關鍵注意
  • 1mpid_ready 必須在 tvbs_page_view 之前推送，否則 GA4 第一筆事件缺 mpid
  • 2Web display_mode 固定 'web'；WebView 必須覆寫為 'app_webview'
  • 3Phase 2 才做：video_start / video_progress / video_complete、share、scroll 等互動事件
